## 2. CI/CD пайплайн

### 2.1. Обзор

Файл: `.github/workflows/ci-cd.yml` (69 строк)

Запускается автоматически при каждом `push` или `pull request` в ветку `main`.

### 2.2. Этапы пайплайна

```
Push в main → Test (21 тест) → Lint (HTML/JS) → Docker Build → Deploy (Pages)
```

| Job | Что делает | Зависимости |
|:---|:---|:---|
| **test** | Node.js 20: запуск 21 теста | — |
| **lint** | DOCTYPE, charset, viewport, node --check | — |
| **docker** | Сборка образа, запуск, curl-проверка | test, lint |
| **deploy** | upload-pages-artifact → deploy-pages | docker |

### 2.3. Конфигурация

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
```

Docker job выполняется только для main (`if: github.ref == 'refs/heads/main'`).

## 4. Процедура поддержки инцидентов

### 4.1. Классификация

| Уровень | Описание | Время реакции | Пример |
|:---|:---|:---|:---|
| 🔴 P1 | Приложение недоступно | 1 час | Падение контейнера |
| 🟡 P2 | Модуль не работает | 4 часа | Ошибка авторизации |
| 🟢 P3 | Визуальный баг | 1 день | Сломалась вёрстка |
| ⚪ P4 | Пожелание | 1 неделя | Новая фича |

### 4.2. Алгоритм обработки

```
Инцидент → Определить приоритет →
  P1: Откат → Диагностика → Hotfix → Деплой
  P2: Hotfix-ветка → Исправление → Тесты → Деплой
  P3-P4: Issue → Исправление → Тесты → Деплой
```

### 4.3. Команды диагностики

| Ситуация | Команда |
|:---|:---|
| Логи контейнера | `docker logs fitclub-app` |
| Статус контейнера | `docker inspect fitclub-app` |
| Перезапуск | `docker-compose restart` |
| Откат | `git revert HEAD && git push` |
| Тесты | `node tests/run-tests.js` |
